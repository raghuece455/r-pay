Building R-Pay: A UPI-Style Payment App with Claude Code Agents

How we used Claude Code like a senior engineering partner to design, build, and verify a payment sandbox — without pretending payment systems are simple.

Read time: ~13 minutes

The most dangerous prompt in software

Build me a payment app.

It sounds productive. It is also how you get a beautiful demo with missing idempotency, no audit trail, unclear payment states, and a "success" screen that lies to the user.

Payment systems are genuinely hard. Not because the UI is hard. Because the edge cases are invisible: what happens when the user taps Pay twice? What happens when the network drops between the request and the confirmation? What does "success" actually mean if the payment network has not confirmed it?

So for R-Pay — a fictional UPI-style payment sandbox inspired by common Indian payment app patterns — we did something different. We did not ask Claude Code to magically build everything in one shot. We asked it to slow down, interview us, write useful design docs, design the state machine, implement one vertical slice, and prove it worked.

That is the difference between using Claude as a chatbot and using Claude Code as an agentic coding partner.

First: what is Claude Code and what makes it an "agent"?

Claude Code is Anthropic's CLI tool for working with Claude inside your codebase. But calling it a "CLI" undersells it.

What makes Claude Code an agent is that it can:

  • Gather context — read your files, understand your architecture, look at test results
  • Choose actions — decide which files to edit, which tests to run, in what order
  • Use tools — actually make edits, run commands, check output
  • Iterate — try something, see what happens, adjust

A chatbot answers questions. An agent takes actions toward a goal. Claude Code is the second one.

This matters for payment code because the goal is not just "write code that compiles." The goal is "write code that handles duplicate requests correctly, validates state transitions, and preserves an audit trail." Those are correctness goals that require reading tests, running them, and iterating.

What is R-Pay?

R-Pay (also called Raghu's Pay) is a fictional UPI-style payment sandbox for India. It is not a real payment system. It does not connect to:

  • Real UPI rails or NPCI systems
  • Real banks or PSPs
  • Real payment gateways
  • Real money

Instead, it uses a local UPI Switch Simulator — a sandbox service that mimics how a payment network would behave. This lets us practice the engineering shape of a real payment system without crossing financial or legal boundaries.

R-Pay has two surfaces:

Step 1: Make Claude interview you first

The first prompt should not be an implementation prompt.

It should make Claude ask questions. Here is the prompt we used:

You are helping me build R-Pay, a fictional UPI-style payment sandbox for India.

Before writing code, interview me like a senior product engineer.

Ask practical questions about:
- user flows
- payment states
- idempotency
- UPI Switch Simulator behavior
- transaction history
- audit logs
- support flows
- observability

Safety boundary:
- no real UPI APIs
- no bank APIs
- no real money movement
- use only a local payment network simulator

After the interview, summarize the MVP scope and risks.

This prompt does two things. It tells Claude the domain is payment-like, which raises the safety bar. And it prevents the classic "giant implementation first, requirements later" mistake.

Claude came back with questions like: "What happens if the same payment request is sent twice?" and "Who must approve if a payment gets stuck?" Those questions forced us to define idempotency and audit requirements before writing a single line of code.

Step 2: Write useful design docs, not museum docs

After the interview, we asked Claude to generate design documents. Not long documents nobody reads — short, practical ones that the code actually references.

The result lives in docs/architecture/:

  • SPEC.md — product scope and safety boundary
  • ARCHITECTURE.md — system components and how they connect
  • DATA_MODEL.md — Prisma schema rationale
  • API_DESIGN.md — endpoint contracts
  • PAYMENT_STATE_MACHINE.md — states and allowed transitions
  • OBSERVABILITY_PLAN.md — what metrics matter and why

Good docs are not ceremony. They are memory for humans and for Claude.

Every Claude Code session starts fresh. When Claude reads ARCHITECTURE.md at the start of a new session, it understands the system without you re-explaining it. That matters a lot over a long project.

Step 3: Design the state machine before the screen

The payment status screen looks simple. It shows "Success" or "Failed." But what that screen shows depends on a backend state machine that must be exactly right.

R-Pay's payment states are:

CREATED → PROCESSING → SUCCESS
                     → FAILED
                     → PENDING → SUCCESS (later confirmation)
                              → FAILED
                              → TIMED_OUT → RECONCILED

The most important rule in the whole codebase is blunt:

R-Pay must not mark a payment SUCCESS unless the payment network simulator confirms it.

This is enforced in code, not just in policy. In packages/shared/src/state-machine.ts, every transition is validated. In the payment service, there is an explicit guard:

// packages/shared/src/state-machine.ts
assertPaymentTransition(fromStatus, toStatus);

// apps/api/src/services/payment-service.ts
if (params.to === "SUCCESS" && !params.confirmedByMock && !payment.confirmedByMock) {
  throw new HttpError(409, "Cannot mark payment SUCCESS without mock UPI confirmation");
}

This is not fancy code. It is adult supervision for a dangerous domain.

We asked Claude to review the state machine before implementing any UI:

Review the R-Pay payment state machine.

Check:
- every allowed transition
- every rejected transition
- whether SUCCESS requires payment network confirmation
- whether PENDING can be reconciled safely
- whether tests cover invalid transitions

Do not change behavior yet.
Return findings first, then suggest small patches.

Claude returned a list of gaps. We fixed them before touching the UI.

Step 4: Build one vertical slice

The first useful slice was end-to-end but narrow:

  1. User enters receiver UPI ID and amount
  2. Web app sends POST /api/payments with an Idempotency-Key header
  3. API creates the payment in state CREATED
  4. API transitions to PROCESSING
  5. API calls the UPI Switch Simulator
  6. Simulator returns SUCCESS, FAILED, or PENDING
  7. API records the attempt and transitions state
  8. User sees status screen

Here is the actual request flow in the Express API (apps/api/src/app.ts):

app.post("/api/payments", asyncHandler(async (request, response) => {
  const idempotencyKey = request.header("Idempotency-Key") ?? undefined;
  const result = await createPayment(request.body, idempotencyKey, config.mockUpiMode);
  response.status(result.duplicate ? 200 : 201).json(result);
}));

Notice two things. First, the idempotency key comes from the request header — it is required. Second, the response status is 200 for a duplicate (returning the existing payment) and 201 for a new one. That distinction matters for clients that need to know whether their request created something new.

The payment form: receiver UPI ID, amount chips, and note field. Behind this screen is a state machine, idempotency check, and audit log.

Why idempotency is not optional in payments

Mobile networks retry. Users tap buttons twice. API clients time out and try again. If a duplicate request creates a duplicate payment, your system is not "eventually consistent." It is wrong.

ProblemWithout idempotencyWith idempotency
User taps Pay twiceTwo payment records, double chargeOne payment, one idempotent replay
Network timeout, client retriesPossible duplicate paymentClient safely retries, gets same result
Support investigationAmbiguous duplicate recordsOne stable reference ID

In R-Pay, every payment requires an Idempotency-Key header. If the same user sends the same key twice, the API returns the original payment and logs an idempotent replay event. No duplicate transaction. No silent failure.

// apps/api/src/services/payment-service.ts
const existing = await prisma.payment.findUnique({
  where: { userId_idempotencyKey: { userId: user.id, idempotencyKey } }
});

if (existing) {
  await prisma.auditLog.create({
    data: {
      paymentId: existing.id,
      event: "payment.idempotent_replay",
      actor: "api",
      metadata: { idempotencyKey, reason: "DUPLICATE_REQUEST" }
    }
  });
  return { payment: toApiPayment(existing), duplicate: true };
}

Audit logs are a product feature

Audit logs sound internal until a user opens support and says: "My payment is stuck. What happened?"

R-Pay stores a full event timeline for every payment:

  • payment.created — when the payment was created and by whom
  • payment.state_changed — from which state to which, with reason
  • payment.idempotent_replay — if the same request was sent twice
  • payment.pending_retry_observed — if the reconciler checked again

This gives the support team (and Claude during an incident) a clear timeline. It is how a later investigation can say "the payment was created at 12:07, transitioned to PENDING at 12:07:03, and the reconciler checked it 14 times before giving up."

Transaction history: every payment has a reference ID, idempotency key, attempt count, and audit timeline you can inspect.

CLAUDE.md: the most important file in the repo

Every Claude Code session starts fresh. It does not remember what you told it last time. Without a way to persist project rules, you end up repeating the same safety instructions in every prompt.

The solution is CLAUDE.md. It is a Markdown file in your repo root that Claude Code reads at the start of every session. It is project memory that you write on purpose.

R-Pay's CLAUDE.md tells Claude:

  • R-Pay is fictional and sandbox-only — no real payment APIs
  • Payment code is high-risk — treat it carefully
  • Every payment needs an idempotency key
  • SUCCESS requires payment network simulator confirmation
  • Audit logs must never be deleted
  • Production-like deployment or rollback needs human approval
  • Run tests after changing payment state logic

The point is not to make Claude perfect. The point is to reduce repeat mistakes. A rule in CLAUDE.md is a rule Claude sees every time, not just when you remember to say it.

Here is the pattern to make your own:

Create a CLAUDE.md for this repo.

Include:
- product purpose
- repo structure
- setup commands
- safety boundary
- payment state machine rules
- idempotency requirements
- audit log requirements
- incident response rules
- testing expectations

Keep it short enough that Claude will actually follow it.
Use concrete rules, not vague advice.

The build loop that worked

The useful workflow was not "ask Claude to build everything." It was a short cycle:

Idea
  → Product interview (Claude asks questions)
  → Design docs (Claude writes, you review)
  → State machine (Claude reviews, you approve)
  → Small vertical slice (Claude implements)
  → Tests (Claude generates and runs)
  → Update CLAUDE.md (capture new rules)
  → Next slice

Claude Code is strongest when it can gather context, take action, and verify results in a tight loop. That means reading files, editing TypeScript, running tests, and seeing output. The key move is not "write code faster." It is "make the repo teach Claude how to behave."

What Claude should never do in payment code

Speed without judgment is dangerous in payment code. Be explicit about the limits:

  • Do not skip idempotency
  • Do not mark a payment SUCCESS without payment network confirmation
  • Do not delete audit logs
  • Do not change the payment state machine without tests
  • Do not connect to real payment APIs in this sandbox
  • Do not hide payment failures behind happy UI copy
  • Do not "fix" stuck payments by editing records directly

These rules live in CLAUDE.md. They are not just suggestions — they are instructions Claude sees every session.

Key takeaways from Part 1

  • Interview before implementing. The first Claude prompt should make Claude ask questions, not write code.
  • Design docs are Claude's memory. They reduce the time spent re-explaining the system.
  • State machines belong in shared packages. packages/shared/src/state-machine.ts enforces rules everywhere.
  • Idempotency is mandatory. No exceptions for payment APIs.
  • Audit logs are a product feature. They answer support questions and enable incident investigation.
  • CLAUDE.md is project memory. Every session starts with the same rules.
  • Build one slice at a time. Correctness over surface area.
