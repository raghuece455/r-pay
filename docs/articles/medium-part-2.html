The problem with "just ask Claude" during an incident

Imagine the pager fires at 12:07 AM.

Payment success rate is dropping. Latency is spiking. The database pool is turning red. A release went out thirty minutes ago, but nobody remembers exactly what it changed.

You open Claude Code and type:

Please investigate.

That is too late.

During an incident, you do not want to teach Claude what R-Pay is, which services exist, which actions are dangerous, how payment states work, or who must approve rollback. You want Claude to already know.

Part 1 gave us a working R-Pay payment sandbox. Part 2 turns that codebase into an incident-ready system.

Claude Code's memory model: how it actually works

To use Claude Code well during incidents, you need to understand how it holds information. There are several layers, and each has a different job.

Layer 1: The current session context

Claude Code holds everything from the current conversation in its context window. This is temporary. When you close the session, it is gone.

This is enough for a single task — "edit this function" — but not for recurring rules like "always require human approval for rollback."

Layer 2: CLAUDE.md — project memory you write on purpose

CLAUDE.md is a Markdown file at your repo root. Claude Code reads it at the start of every session. It is how you persist rules across sessions without repeating yourself in every prompt.

For R-Pay, CLAUDE.md contains the rules that must never be forgotten:

## Safety Boundary
- Do not use real UPI APIs, bank APIs, or NPCI credentials
- Payment code is high-risk
- Every payment requires an idempotency key
- Do not mark a payment SUCCESS without simulator confirmation
- Do not delete audit logs
- Production-like deployment or rollback requires human approval

## Engineering Rules
- Shared payment states live in packages/shared
- API routes live in apps/api
- Worker retry behavior lives in apps/worker
- Run tests after changing payment state logic

These rules show up in every session without you typing them.

Layer 3: Auto memory — what Claude discovers itself

Auto memory is memory Claude Code can write for itself when it learns something useful. For example, if Claude discovers that your repo uses pnpm (not npm), it can save that so it does not have to re-discover it.

For high-risk rules in payment systems, keep them in CLAUDE.md where you can review them. Use auto memory for workflow friction (like "this test needs a specific environment variable"), not safety rules.

Layer 4: Skills — reusable workflows you define

A skill is a reusable task instruction. Instead of typing the same multi-step prompt over and over, you define a skill once and invoke it by name.

For example, an investigate-payment-incident skill would tell Claude to:

  1. Read CLAUDE.md and the payment runbook
  2. Check current metrics and active incidents
  3. Inspect recent deployments
  4. Inspect logs and traces
  5. Form hypotheses with supporting evidence
  6. Recommend mitigation options
  7. Flag which actions require human approval

You run that skill with one command at 2 AM instead of typing all seven steps while panicking.

Layer 5: Hooks — automatic safety gates

Hooks are commands that run automatically at lifecycle events. Unlike CLAUDE.md rules (which Claude reads but could theoretically ignore), hooks run regardless.

For R-Pay, useful hooks would include:

  • Pre-task hook: run pnpm test before marking anything complete
  • State machine change hook: require tests when state-machine.ts is modified
  • Deploy hook: block production-like actions without an explicit approval flag
  • Audit log hook: warn if code changes would delete from the audit log table

Hooks are where you stop relying on memory alone. They make safety automatic.

What is a subagent and when do you use one?

A subagent is a focused Claude Code helper with its own context and a narrow job. When you ask Claude to use a subagent, it spins up a focused session that investigates one specific area and reports back.

Think of it like having a team where each person specialises:

  • log-analyst — searches logs for patterns, correlates trace IDs, builds a timeline of events
  • trace-analyst — reads distributed traces, identifies slow spans, maps pressure to services
  • reliability-engineer — analyses retry behavior, queue depth, database pool saturation
  • release-manager — correlates incidents with deployments, prepares rollback plans
  • incident-commander — coordinates the investigation, synthesises findings, recommends actions
  • postmortem-writer — drafts RCA from timeline and evidence

Each subagent has a safety boundary explicitly written into its definition. The log analyst, for example, must not recommend direct database edits. The release manager must require human approval before any deployment action. These rules are not optional hints — they are part of the subagent's instructions.

Why not just use one big Claude session?

You can. For simple incidents, one Claude session is faster and cheaper. Subagents cost more context.

Use subagents when:

  • The investigation task is large enough to fill the context window
  • You need a specialist to focus on one area without distraction
  • You want to run parallel investigations simultaneously

For R-Pay's Midnight Retry Storm, the log analyst, trace analyst, and reliability engineer can investigate in parallel. Each brings back independent findings. That is faster than one session doing all three jobs serially.

Subagents vs agent teams: what's the difference?

Question Subagent Agent team
How many Claude sessions? One main + focused helpers Multiple independent sessions
Communication style Subagent reports back to you Teammates can coordinate with each other
Best for Focused investigation or review Parallel exploration with multiple perspectives
Cost and overhead Lower Higher
R-Pay example Ask log-analyst to summarise timeout logs Have reliability, release, and test teammates compare rollback vs hotfix

Use subagents when you need focused workers. Use agent teams when the problem benefits from genuine disagreement between perspectives.

Do not use agent teams for every task. If the next step is obvious, a single Claude Code session plus tests is faster and cheaper.

Where Agent SDK and Managed Agents fit

The R-Pay demo does not call a real Claude API. Its AI incident analysis and RCA panels are generated from local templates. That is intentional for a local sandbox.

If you wanted to build a real internal engineering assistant on top of R-Pay's architecture, here is where the bigger tools would come in:

Option Use when R-Pay example
Agent SDK You want to run a custom agent workflow in your own service or CI job A scheduled agent that reads recent deployments and opens a PR with runbook updates
Managed Agents You want Anthropic to host a long-running agent with persistent environment A production on-call assistant that watches incident tickets and prepares evidence summaries with approval gates

The safety rule stays the same regardless of which tool you use:

More automation does not remove human approval for risky payment actions.

IncidentDesk: the shared evidence board

IncidentDesk is the ops surface of R-Pay. It is not just a dashboard — it is the place where humans and agents share the same evidence.

IncidentDesk payment health dashboard: success rate, P95 latency, queue depth, DB pool usage, and retry rate — the metrics that tell the incident story fast.

The dashboard shows:

  • Payment success rate and failure rate
  • P95 latency (milliseconds to seconds is a clear signal)
  • Pending payment count
  • Status reconciler queue depth
  • Database connection pool usage
  • Payment network simulator latency
  • Retry rate (healthy vs unsafe)
  • Service health for all five services

These metrics matter together, not individually. A retry rate spike means nothing if you do not also see where queue depth and DB pool usage went. The dashboard puts all of them on one screen so the incident story is visible at a glance.

Logs and traces: finding the evidence

Logs and traces browser: filters, trace IDs, and expanded log entries help Claude's log-analyst subagent build a timeline from noisy signals.

During an incident, a log analyst should not just dump logs. It should answer:

  • What changed compared to the normal baseline?
  • Which service is generating the most noise?
  • Which trace IDs connect across services?
  • Are the failures user-facing or background-only?
  • What evidence supports or contradicts the current hypothesis?

When the Midnight Retry Storm hits, the key log entries look like this:

00:10:14Z WARN  status-reconciler retry_storm.detected
  retryRatePerMin=16500 queueDepth=98200 dbPoolUsage=100

00:10:35Z ERROR payment-api db_pool.saturated
  p95LatencyMs=4800 pendingPayments=3296

00:11:02Z WARN  upi-switch-simulator latency.threshold_exceeded
  latencyMs=5100 mode=slow

Three log lines. Three different services. One incident. The log analyst's job is to connect those dots before the team spends an hour looking in the wrong place.

Deployment history: the fastest path to a hypothesis

Deployment history: the suspicious worker release is immediately visible next to the incident start time.

One of the most reliable patterns in production incidents is: something changed recently. The deployment history page makes that visible immediately.

In the Midnight Retry Storm, the suspicious release is:

rpay-worker-2026.05.28-001
Service: status-reconciler
Change: Changed status reconciler retry polling behavior
Risk: High
Linked incident: The Midnight Retry Storm

That link between deployment and incident is the first hypothesis the release-manager subagent would surface. It takes a few seconds to see on the dashboard and would have taken thirty minutes to find by reading git history in a panic.

Runbooks: the incident playbook that actually works at 2 AM

Runbook viewer: step-by-step response procedure that works when your brain is running at half speed.

A good runbook is short and concrete. R-Pay's payment incident runbook:

  1. Confirm the alert scope and declare the incident severity
  2. Check success rate, latency, pending count, queue depth, and DB pool in IncidentDesk
  3. Check recent deployments — look for changes in the last two hours
  4. Check payment network latency
  5. Check status reconciler queue depth and retry rate
  6. Check database connection pool saturation
  7. Decide: rollback vs feature flag vs hotfix
  8. Get human approval for any action
  9. Monitor recovery for 10 stable minutes
  10. Generate RCA draft from timeline and evidence

Ten steps. No fluff. Readable when you have been awake since midnight.

The AI incident analysis panel

IncidentDesk includes an AI incident analysis panel. In the local sandbox, this is generated from templates and simulated evidence — it does not call a real Claude API.

But the panel structure matches how a real incident commander subagent should report:

  • Confidence — how certain is the hypothesis?
  • Current hypothesis — what do we think is happening and why?
  • Evidence — what facts support the hypothesis?
  • Recommended mitigation — what should we do right now?
  • Human approval requirement — what needs a human to sign off?

The last point is the most important. Claude is useful for gathering context and forming hypotheses. The human owns the decision and the accountability.

The rules vs skills vs hooks distinction, simply

Developers often mix these up. Here is the practical version:

ConceptWhat it isR-Pay example
Rule (in CLAUDE.md) A persistent fact Claude sees every session "Never mark SUCCESS without payment network confirmation"
Skill A reusable multi-step workflow you define once and invoke by name /investigate-payment-incident — gathers metrics, logs, deployments, forms hypotheses
Hook An automatic gate that runs regardless of what Claude was planning to do Run pnpm test before task completion; block deploy without approval flag

If Claude needs to remember a fact, put it in CLAUDE.md.

If Claude needs to follow a multi-step process repeatedly, make it a skill.

If something must run regardless of Claude's judgment, make it a hook.

Common mistake: teaching Claude only through chat

If you paste the same safety rule into every prompt, the project is telling you something. That rule belongs in CLAUDE.md, a skill, or a hook.

Bad pattern:

Remember again that payment SUCCESS needs confirmation from the switch simulator...

Better pattern:

  • Put the rule in CLAUDE.md
  • Add a test that proves the rule is enforced
  • Add a hook that detects state machine changes
  • Ask a security-reviewer subagent to check payment safety

Chat context is temporary. Repo memory is durable. The difference matters at 2 AM when you open a new session and cannot remember what you already told Claude.

Checklist: is your system incident-ready?

  • ☐ CLAUDE.md explains the product boundary and all payment rules
  • ☐ Skills exist for repeat incident workflows (investigate, generate-rca, prepare-rollback)
  • ☐ Hooks protect high-risk operations (tests required, deploy blocked without approval)
  • ☐ Subagents have narrow roles and explicit safety rules
  • ☐ Agent teams are reserved for problems that need parallel perspectives
  • ☐ IncidentDesk shows metrics, logs, traces, deployments, and runbooks in one place
  • ☐ Rollback and deployment actions require human approval — in the UI, not just in policy
  • ☐ RCA drafting is based on evidence, not guesswork
