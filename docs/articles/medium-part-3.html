The first prompt: ask for evidence, not a fix

The most important discipline in agent-assisted incident response is this: the first prompt should ask for evidence, not a solution.

We have an active R-Pay payment incident.

Read CLAUDE.md, the architecture docs, the payment runbook,
metrics, logs, traces, deployments, and worker retry logic.

Return:
- current customer impact
- affected services
- strongest hypotheses with evidence
- what is unknown
- immediate mitigation options
- which actions require human approval

Safety:
- R-Pay is sandbox-only
- do not edit payment records directly
- do not mark PENDING payments SUCCESS
- do not delete audit logs
- do not deploy or rollback without approval

Notice what this prompt does not ask for: a fix. It asks Claude to gather, synthesise, and present. The humans then decide.

What each subagent found

The log analyst

Logs browser: the retry storm pattern is visible from three log entries across three different services.

The log analyst subagent filtered the noise and surfaced three key log events:

  retryRatePerMin=16500 queueDepth=98200 dbPoolUsage=100

00:10:35Z ERROR payment-api db_pool.saturated
  p95LatencyMs=4800 pendingPayments=3296

00:11:02Z WARN  upi-switch-simulator latency.threshold_exceeded
  latencyMs=5100 mode=slow

The log analyst's report:

"Top pattern: the status reconciler is retrying at 16,500/min — 118× above the normal budget of 140/min. The retry spike started at 00:08:31Z, three minutes after the latest deployment at 00:05:44Z. DB pool saturation appears secondary to the retry pressure. UPI switch simulator latency is elevated but is not the initiating cause — it worsened the situation by making each retry take longer. What is unknown: whether any PENDING payments have already exceeded their reconciliation TTL."

That is not "root cause found." That is a clear pattern, a timeline, and an honest uncertainty. That is what a good subagent report looks like.

The trace analyst

The trace analyst read the distributed trace spans and found the critical shape of the problem:

payment-api.createPayment         → 280 ms (normal)
  ├─ db.payment.insert            → 12 ms
  ├─ db.auditLog.insert           → 8 ms
  ├─ upi-switch-simulator.pay     → 180 ms
  ├─ db.paymentAttempt.insert     → 11 ms
  └─ db.payment.update            → 69 ms (high — pool contention)

status-reconciler.retryLoop       → 4,820 ms (17× slower than normal)
  ├─ db.pendingPayments.findMany  → 2,100 ms (pool contention)
  ├─ upi-switch-simulator.status  → 1,900 ms (slow)
  ├─ db.paymentAttempt.insert     → 380 ms
  ├─ db.payment.update            → 440 ms
  └─ [repeating every 1,000 ms with no backoff]

The trace tells the whole story. The payment creation API itself is mostly healthy. The problem is the reconciler running every second, consuming database connections that the API also needs.

The code investigator

The code investigator read the worker source and the shared retry logic:

// apps/worker/src/index.ts (buggy mode — what shipped in the bad release)
// STATUS_RECONCILER_MODE = "buggy"
// Fixed 1-second polling. No exponential backoff. No jitter. No retry budget.
// Every pending payment fires a new DB query every second.

// packages/shared/src/retry.ts (healthy mode — what it should be)
// Exponential backoff: 2^attempt * baseDelayMs
// Jitter: ±30% randomisation to prevent thundering herd
// Retry budget: max 3 attempts per payment per window
// Circuit breaker: pause reconciliation if UPI switch is slow

The reliability engineer

The reliability engineer's job was to connect all the dots into one coherent failure loop:

UPI Switch Simulator becomes slow (latency: 310 ms → 5,100 ms)
        │
        ▼
Status Reconciler polls every 1 second (no backoff, no jitter)
        │
        ▼
Each of 3,296 PENDING payments generates a DB read + write every second
        │
        ▼
Queue depth: 1,200 → 98,200
        │
        ▼
DB connection pool reaches 100% — no connections left for Payment API
        │
        ▼
Payment API latency: 280 ms → 4,800 ms
        │
        ▼
Users see new payments stuck in PROCESSING or PENDING
        │
        ▼
More PENDING payments → more reconciler pressure → loop continues

This is a classic cascading failure: one service's poor behaviour under pressure overloads a shared resource (the database pool), which degrades a completely different service (the payment API) that happens to share that resource.

The critical insight from the reliability engineer: "The database was not the root cause. It was the victim. Fixing the database connection settings would only buy time. The root cause is the retry behavior."

"Candidate for rollback: rpay-worker-2026.05.28-001. Evidence strength: high. Rollback risk: low — the worker is a separate process with no schema changes. The payment creation API can stay online during rollback. Recommend rollback before any code changes."

Rollback vs hotfix: making the decision

At 12:17 AM, the team had two options:

Option Pros Cons Decision
Roll back the worker Fast. Low code risk. Targets the exact change. No new code to write at midnight. Does not add the guardrails that should have prevented this. ✓ Do this first
Hotfix retry logic Permanent fix. Adds circuit breaker and backoff. Risky to write and deploy new code during an active P1. Harder to review under pressure. Do this after recovery

The decision tree looks like this:

P1 payment incident
        │
        ▼
Does evidence point to a recent worker release?
  YES → Can rollback isolate the worker without corrupting payment state?
          YES → Request human approval → Rollback → Monitor
          NO  → Prepare targeted hotfix
  NO  → Continue investigation

In R-Pay, rolling back the worker does not affect payment state at all. The worker only does reconciliation — it reads and updates existing PENDING payments. Rolling back to the previous version restores safe retry behavior without touching any in-flight payments.

The human approval gate

Rollback is a production-like action. Even in this local simulation, the UI requires an explicit approval step.

The modal copy makes the boundary clear:

This pattern matters. In a real system, the approval text would be different — it might require a Slack approval message from a manager, or a two-person rule for financial system changes. But the shape is the same: Claude can prepare the action, explain the risk, and show the blast radius. Humans decide whether to run it.

The permanent fix

After recovery, the code investigator helped draft the permanent fix. The key changes from packages/shared/src/retry.ts:

// BEFORE (buggy mode — what shipped)
// Fixed 1-second polling, no backoff, no jitter, no budget, no circuit breaker

// AFTER (fixed mode — what ships in the permanent fix)
function shouldRetry(attempt: number, lastLatencyMs: number): RetryDecision {
  // Exponential backoff: 1s → 2s → 4s → 8s → 16s
  const backoffMs = Math.min(BASE_DELAY_MS * Math.pow(2, attempt), MAX_DELAY_MS);

  // Jitter: ±30% to prevent thundering herd when many payments retry together
  const jitter = backoffMs * 0.3 * (Math.random() * 2 - 1);
  const delayMs = backoffMs + jitter;

  // Retry budget: no payment gets more than 3 attempts per reconciliation window
  if (attempt >= MAX_ATTEMPTS) return { shouldRetry: false, reason: "BUDGET_EXHAUSTED" };

  // Circuit breaker: if the UPI switch is slow, pause all reconciliation temporarily
  if (lastLatencyMs > CIRCUIT_BREAKER_THRESHOLD_MS) {
    return { shouldRetry: false, reason: "CIRCUIT_OPEN", pauseMs: CIRCUIT_PAUSE_MS };
  }

  return { shouldRetry: true, delayMs };
}

The key principle: the reconciler should back off under pressure, not increase pressure. Fixed 1-second polling does the opposite — it hammers harder exactly when the system is already struggling.

The permanent fix prompt:

Create the permanent R-Pay retry fix.

Requirements:
- restore exponential backoff
- add jitter (±30%)
- enforce retry budget (max 3 attempts per payment per window)
- add circuit breaker when payment network simulator is slow
- preserve idempotency across retries
- keep pending payments safe — do not mark SUCCESS without confirmation
- add or update tests for healthy, buggy, and fixed retry modes
- add alert recommendations for retry rate, queue depth, pending age, and DB pool saturation

Return a patch summary and test results.

Why tests missed it

This question matters for the RCA. The unit tests for retry logic existed and passed. How did a broken retry behavior ship?

The answer: it was not a unit bug. It was an interaction bug.

The failure required all of these to be true simultaneously:

  1. UPI Switch Simulator in slow mode
  2. Large number of PENDING payments (3,000+)
  3. Worker retry loop with fixed 1-second interval
  4. Shared database pool between worker and API
  5. No circuit breaker to pause reconciliation under load

Unit tests check one piece at a time. This bug only appeared when all five pieces were in play at once. That requires a load test or an integration test that simulates system pressure — not just function-level tests.

The test additions needed after this incident:

  • Load test: 3,000 PENDING payments + slow UPI simulator + full retry cycle
  • Integration test: confirm retry rate stays within budget under slow payment network conditions
  • Alert test: verify retry rate and queue depth alerts fire at defined thresholds
  • Circuit breaker test: confirm reconciler pauses when simulator latency exceeds threshold

What Claude should never do during a payment incident

These rules live in CLAUDE.md and in every subagent definition. They are not optional:

  • Do not directly edit payment records in the database
  • Do not mark PENDING payments as SUCCESS to "clear the queue"
  • Do not delete audit logs to "clean things up"
  • Do not deploy or rollback without explicit human approval
  • Do not assume root cause from a single log line
  • Do not hide uncertainty in a hypothesis — say what is still unknown
  • Do not ignore user-facing impact while chasing elegant technical solutions
  • Do not bypass the runbook because the dashboard looks obvious

The real lesson

Claude did not save production. That is the wrong lesson.

The right lesson is smaller and stronger:

Claude helped the team gather context, inspect evidence, compare options, prepare safe changes, and draft the RCA. The humans owned the decision, the approval, and the accountability.

That is how AI agents should work near payment systems. Fast hands. Clear evidence. Strong guardrails. Human approval where it matters.

A developer working with Claude on an incident like this does not feel stressed. They feel supported. The agents handle the information gathering — finding the right log lines, correlating the right deployment, reading the retry code, building the timeline. The developers make the calls.

That is the version of agentic AI that is actually useful in production: not autonomous, but deeply collaborative.
